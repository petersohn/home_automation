# Migrate host-side tests from Boost.Test to Googletest

## Goal

Replace Boost.Test as the host-side unit testing framework for this project
with Googletest, with no behavior change to the tests themselves and no change
to device code. The `home_automation_test` binary continues to build via CMake
and the ESP8266 firmware continues to build via `arduino-cli`.

The `bin/operation_tester` CLI binary, the `test/ArduinoJson` and `src/SDS011`
submodules, and the device source tree are explicitly out of scope.

## Background

The current test target links `boost_unit_test_framework` and uses Boost.Test
macros throughout. There are 16 test files in `test/`, totaling 16 source
files that include Boost.Test headers and a `testMain.cpp` that defines
`BOOST_TEST_MODULE HomeAutomationTest`. The Boost.Test idioms in use are:

- `BOOST_AUTO_TEST_SUITE` / `BOOST_AUTO_TEST_SUITE_END` (with nesting)
- `BOOST_AUTO_TEST_CASE`, `BOOST_FIXTURE_TEST_CASE`
- `BOOST_DATA_TEST_CASE_F` with `boost::unit_test::data::make(...)` datasets,
  combined with `*` (Cartesian product) and `+` (concatenation)
- `BOOST_TEST`, `BOOST_TEST_REQUIRE`, `BOOST_TEST_MESSAGE`,
  `BOOST_TEST_CONTEXT`, `BOOST_REQUIRE`, `BOOST_REQUIRE_EQUAL`,
  `BOOST_REQUIRE_NO_THROW`, `BOOST_FAIL`,
  `BOOST_CHECK_EQUAL_COLLECTIONS`
- `boost::test_tools::tolerance(...)` for floating-point comparisons
- A custom `std::boost_test_print_type` overload for
  `std::optional<std::vector<std::string>>`
- Free `operator<<` overloads for `StatusMessage`, `AvailabilityMessage`,
  `ConnectionAttempt`, and `OperationTestSample*` structs in
  `OperationParserTest.cpp`

CMake integration currently uses `find_package(Boost COMPONENTS
unit_test_framework REQUIRED)` and links `boost_unit_test_framework`.

## Approach

Add Googletest as a git submodule at `test/googletest` (pinned to release tag
`v1.15.2`), compile it via `add_subdirectory(test/googletest)`, and rewrite
all 16 test files in a single change. The migration is one-shot — there is no
intermediate state in which Boost and gtest coexist.

This mirrors the existing submodule pattern used for `test/ArduinoJson` and
`src/SDS011` and avoids the well-known "wrong gtest version" footgun that
affects `find_package(GTest)` on Debian/Ubuntu.

## Submodule setup

- `git submodule add https://github.com/google/googletest.git test/googletest`
  at tag `v1.15.2`.
- Add the entry to `.gitmodules`:

  ```
  [submodule "test/googletest"]
      path = test/googletest
      url = https://github.com/google/googletest.git
  ```

- `git submodule update --init --recursive` after the add.

## CMake changes

`CMakeLists.txt`:

- Remove `find_package(Boost COMPONENTS unit_test_framework REQUIRED)`.
- Remove `target_include_directories(home_automation_test PRIVATE
  ${Boost_INCLUDE_DIRS})`.
- Remove `target_link_libraries(home_automation_test boost_unit_test_framework)`.
- Add `add_subdirectory(test/googletest)` after the `file(GLOB ...)`
  declarations.
- Change the `home_automation_test` link line to
  `target_link_libraries(home_automation_test gtest gtest_main)`.
- The `testMain.cpp` is no longer globbed in once it is deleted (gtest's
  `gtest_main` provides `main()`).

The `operation_tester` target is left as-is; it does not use Boost.

`build/compile_commands.json` is regenerated by `cmake` and the project-root
symlink picks up the new file automatically.

## Test file rewrites

For each `.cpp` under `test/`:

1. Replace `<boost/test/unit_test.hpp>` and any other Boost.Test includes
   (`<boost/test/data/test_case.hpp>`,
   `<boost/test/unit_test_log.hpp>`, etc.) with `<gtest/gtest.h>`. If a file
   only used Boost headers for assertion macros that are no longer needed,
   remove the include entirely.
2. Apply the assertion translation rules below.
3. Apply the suite/case/fixture translation rules below.
4. Keep the file naming, helper classes (`EspTestBase`, `InterfaceTestBase`,
   `Fake*`, `TestStream`, `LogExpectation`, `DummyBackoff`), and the build
   glob unchanged.

The `LogExpectation` helper at `test/LogExpectation.cpp` /
`test/LogExpectation.hpp` includes `<boost/test/unit_test.hpp>` to use
`BOOST_REQUIRE`. The include is replaced with `<gtest/gtest.h>` and the
`BOOST_REQUIRE(...)` is changed to `ASSERT_TRUE(...)` (or
`EXPECT_TRUE(...)`, matching how `LogExpectation` reports failure).

The `Fake*` and `TestStream` translation units include Boost headers only for
the `BOOST_TEST_DYN_LINK` macro chain; those includes are removed because
gtest does not need them.

## Suite / case / fixture translation

| Boost.Test | Googletest |
|---|---|
| `BOOST_AUTO_TEST_SUITE(Outer)` / `BOOST_AUTO_TEST_SUITE_END` | `TEST(Outer, …)` / `TEST_F(Outer, …)` at namespace scope |
| Nested `BOOST_AUTO_TEST_SUITE(Inner)` inside `Outer` | Flatten into `TEST(Outer, Inner)`, or use `TEST(Outer_Inner, …)` if flattening would cause name collisions. The current code has a single nested suite (`CollectionTest/FindValueTest`); it is flattened to `TEST(CollectionFindValueTest, …)`. |
| `BOOST_AUTO_TEST_CASE(Name)` | `TEST(SuiteName, Name)` |
| `BOOST_FIXTURE_TEST_CASE(Name, Fixture)` | `TEST_F(Fixture, Name)` |
| `BOOST_DATA_TEST_CASE_F(Fixture, Name, dataset, params)` | `TEST_P(Fixture, Name)` plus an `INSTANTIATE_TEST_SUITE_P` declaration with the dataset. |

The Boost.Test-to-Googletest mapping preserves test naming: every
`BOOST_AUTO_TEST_SUITE(name)` becomes the gtest suite (first argument to
`TEST` / `TEST_F` / `TEST_P`). When a fixture class is the same name as the
outer Boost suite (the common case here — e.g. `CoverTest` fixture and
`BOOST_AUTO_TEST_SUITE(CoverTest)`), the gtest suite is the fixture class
name.

## Parameterized tests

- `boost::unit_test::data::make({10, 50, 100})` → `testing::Values(10, 50, 100)`
  (or `testing::ValuesIn(std::vector{10, 50, 100})`).
- `boost::unit_test::data::make(arr)` where `arr` is a C-array →
  `testing::ValuesIn(arr)`.
- `boost::unit_test::data::make({false, true})` → `testing::Bool()`.
- `a * b` (Cartesian product) → `testing::Combine(testing::ValuesIn(a),
  testing::ValuesIn(b), …)`.
- `a + b` (concatenation) → concatenate the two datasets into a single
  parameter vector and use a **single** `INSTANTIATE_TEST_SUITE_P` with
  that vector. Two `INSTANTIATE_TEST_SUITE_P` calls with the same
  instance-name prefix on the same fixture cause redefinition errors
  because gtest uses the prefix in the generated internal symbol names.
  Use one `INSTANTIATE_TEST_SUITE_P` per `TEST_P`, each with a unique
  prefix that names the test it parameterizes.
- Datasets declared as `const auto foo = ...;` in an anonymous namespace at
  file scope stay in the same anonymous namespace.
- Where the default test-suffix `/0`, `/1`, … is too terse, a custom
  `testing::TestParamInfo<...> -> std::string` lambda is passed as the
  fourth argument to `INSTANTIATE_TEST_SUITE_P` to give the
  combination a readable name.

Concrete rewrites:

`test/CoverTest.cpp:685`:
```
const auto params2NoPositionSensor =
    delays2 * latchings * boost::unit_test::data::make({false, true}) *
    boost::unit_test::data::make({false, true}) *
    boost::unit_test::data::make({false, true});
```
becomes:
```
INSTANTIATE_TEST_SUITE_P(
    Cover, CoverTest,
    testing::Combine(
        testing::ValuesIn(delays2),
        testing::ValuesIn(latchings),
        testing::Bool(), testing::Bool(), testing::Bool()));
```

`test/OperationParserTest.cpp:429`:
```
boost::unit_test::data::make(stringOperations) + comparisons
```
becomes a single `INSTANTIATE_TEST_SUITE_P` against the concatenated
vector:

```cpp
std::vector<OperationParserTestParam> params = toParamVector(stringOperations);
auto more = toParamVector(comparisons);
params.insert(params.end(), more.begin(), more.end());
INSTANTIATE_TEST_SUITE_P(String, OperationParserTest,
                         testing::ValuesIn(params));
```

Multiple `INSTANTIATE_TEST_SUITE_P` calls against the same fixture must
each have a **unique** instance-name prefix.

## Assertion translation

| Boost.Test | Googletest |
|---|---|
| `BOOST_REQUIRE(cond)` | `ASSERT_TRUE(cond)` |
| `BOOST_TEST(cond)` (non-fatal) | `EXPECT_TRUE(cond)` |
| `BOOST_TEST_REQUIRE(cond)` | `ASSERT_TRUE(cond)` |
| `BOOST_TEST(a == b)` where `a`, `b` are built-in or have `operator<<` | `EXPECT_EQ(a, b)` |
| `BOOST_TEST(a == b)` where `a` or `b` is `std::optional<std::vector<std::string>>` (or another type without `operator<<`) | `EXPECT_TRUE(a == b) << "actual=" << testing::PrintToString(a) << " expected=" << testing::PrintToString(b)` |
| `BOOST_TEST(std::atof(x) == y, boost::test_tools::tolerance(1e-6))` | `EXPECT_NEAR(std::atof(x), y, 1e-6)` (review the two call sites: `OperationParserTest.cpp:67` and `OperationParserTest.cpp:456`. `EXPECT_NEAR` uses an absolute tolerance, which matches the intent of these specific tests.) |
| `BOOST_REQUIRE_EQUAL(a, b)` | `ASSERT_EQ(a, b)` |
| `BOOST_REQUIRE_NO_THROW(expr)` | `EXPECT_NO_THROW(expr)` |
| `BOOST_FAIL(msg)` | `FAIL() << msg` |
| `BOOST_TEST_MESSAGE(msg)` (inside a test body) | `RecordProperty("message", msg)` |
| `BOOST_TEST_MESSAGE(msg)` (inside a non-test helper function) | `std::cout << msg << std::endl;` — `RecordProperty` is not available outside test bodies. This makes the message always-visible instead of gated by Boost's `--log_level`. The four affected call sites are in `FakeEspApi.cpp`, `FakeMqttConnection.cpp`, `FakeRtc.cpp`, and `TestStream.cpp`. |
| `BOOST_TEST_CONTEXT(name) { body; }` | `SCOPED_TRACE(name); body;` (the `SCOPED_TRACE` lives at the same scope as the original `BOOST_TEST_CONTEXT` and applies to the rest of the function / block) |
| `BOOST_CHECK_EQUAL_COLLECTIONS(a_begin, a_end, b_begin, b_end)` | `EXPECT_TRUE(std::equal(a_begin, a_end, b_begin, b_end))` (a small `EXPECT_COLLECTIONS_EQ` helper is introduced in `test/TestHelpers.hpp` to print a one-line diff on failure) |

## Custom printers

The `std::boost_test_print_type` overload for
`std::optional<std::vector<std::string>>` in
`test/AnalogSensorTest.cpp:17` is replaced by a gtest `PrintTo` overload in
the same `std` namespace:

```cpp
namespace std {
void PrintTo(const std::optional<std::vector<std::string>>& v,
             std::ostream* os) {
    if (!v) { *os << "<none>"; return; }
    *os << "{";
    for (size_t i = 0; i < v->size(); ++i) {
        *os << "\"" << (*v)[i] << "\"";
        if (i + 1 < v->size()) *os << ", ";
    }
    *os << "}";
}
}  // namespace std
```

The free `operator<<` overloads for `StatusMessage`, `AvailabilityMessage`,
`ConnectionAttempt`, and the `OperationTestSample*` structs are kept
unchanged — gtest discovers them via the same `operator<<` lookup as Boost.

## Run command

Old:
```
./build/home_automation_test --log_level=test_suite [ -t name ]
```

New:
```
./build/home_automation_test [--gtest_filter=SuiteName.*] [--gtest_color=yes]
```

Running with no arguments lists all tests and runs them, which is the
default. `--gtest_list_tests` prints the names without running.

`AGENTS.md` is updated to reflect the new command. The `-t` flag is replaced
with the gtest filter form in any places it appears.

## Risks and mitigations

1. **Floating-point tolerance semantics differ.** Boost's
   `tolerance(1e-6)` is a relative tolerance; gtest's `EXPECT_NEAR` is an
   absolute tolerance. The two call sites
   (`OperationParserTest.cpp:67`, `OperationParserTest.cpp:456`) are
   reviewed during the rewrite; if the relative semantics are required,
   switch to `EXPECT_DOUBLE_EQ` or a custom `EXPECT_PRED_FORMAT2`
   predicate. If the absolute tolerance is sufficient, keep `EXPECT_NEAR`.
2. **Removed.** (Earlier drafts described a multi-`INSTANTIATE_TEST_SUITE_P`
   pattern for `+` concatenation, but that is not viable: gtest generates
   internal symbol names from `<prefix><fixture>`, so two
   `INSTANTIATE_TEST_SUITE_P` calls with the same prefix on the same
   fixture cause redefinition errors. The correct pattern is to
   concatenate the parameter sets into a single vector and use a single
   `INSTANTIATE_TEST_SUITE_P` with a unique prefix.)
3. **`ASSERT_*` inside loops** (`CoverTest.cpp:219`–`224`) is permitted
   in gtest and aborts the current function (and therefore the current
   parameterized test instance) — matches Boost behavior.
4. **Hidden Boost.Test use in helper translation units** is verified by
   grep before the rewrite; the only Boost.Test artifact in helpers is the
   include line, which is removed or swapped.
5. **Submodule size.** Googletest is roughly 150 MB at the pinned tag.
   Contributors must run `git submodule update --init --recursive` after
   cloning. AGENTS.md is updated to mention this.

## Out of scope

- Adopting gmock matchers in place of the existing `operator==` definitions.
- Splitting, renaming, or removing any tests.
- Migrating `bin/operation_tester.cpp`.
- Switching to `find_package(GTest)` (would defeat the self-contained build
  goal).
